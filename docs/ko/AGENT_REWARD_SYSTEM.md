# 비ML 개발자를 위한 에이전트 보상 시스템 설명

이 문서는 3월 31일 캡스톤 미팅에서 나온 "모델 학습 이후 에이전트를 붙인다"는 아이디어를 비ML 개발자 기준으로 설명합니다.

핵심은 짧게 말하면 다음과 같습니다.

- `CatBoost`는 예측 모델입니다.
- 에이전트는 그 예측을 바탕으로 행동을 결정하는 주체입니다.
- 보상 시스템은 그 행동이 시간이 지나도 좋은 선택이었는지 평가하는 기준입니다.

## 1. 먼저 피해야 할 오해

모델을 학습하면 프로젝트가 끝난다고 생각하기 쉽습니다.

1. 텔레메트리 수집
2. `CatBoost` 학습
3. failure 예측

하지만 여기까지는 예측 계층만 완성된 상태입니다.

교수님 피드백은 그 다음 계층을 요구한 것입니다.

1. 시스템 상태를 관측한다
2. 앞으로 일어날 가능성을 예측한다
3. 어떤 액션을 취할지 결정한다
4. 그 액션의 결과를 측정한다
5. 보상 신호를 이용해 다음 의사결정을 개선한다

즉, 모델은 "이 머신이나 태스크가 위험해 보인다"를 말하고, 에이전트는 "그 위험을 보고 지금 시스템이 무엇을 해야 하는가"를 결정합니다.

## 2. CatBoost 학습 이후 무엇이 달라지는가

학습이 끝난 뒤 `CatBoost`는 더 큰 제어 루프 안의 한 컴포넌트로 봐야 합니다.

예를 들면 다음과 같습니다.

- 입력: 최근 CPU, 메모리, 태스크 이력, 머신 상태
- `CatBoost` 출력: `failure_risk_15m = 0.87`
- 에이전트 입력: 이 `0.87` 위험도와 현재 클러스터 상태
- 에이전트 출력: 다음과 같은 액션 중 하나 선택
  - 아무것도 하지 않음
  - 태스크 이동
  - 배치 지연
  - 특정 노드의 overcommit 완화
  - low-priority 태스크 축출
  - 운영자에게 알림 전송

정리하면 모델은 예측하고, 에이전트는 결정합니다.

## 3. 보상 시스템은 정확히 무엇인가

보상 시스템은 `CatBoost`를 학습할 때 쓰는 일반적인 ML loss와 다릅니다.

`CatBoost` 학습은 보통 다음과 같은 supervised learning 지표를 사용합니다.

- log loss
- AUC
- PR-AUC / average precision

이 지표들이 답하는 질문은 다음입니다.

- "모델이 위험한 케이스를 잘 순위화했는가?"
- "모델이 failure를 정확하게 예측했는가?"

반면 보상 시스템은 다른 질문에 답합니다.

- "에이전트가 선택한 액션이 실제 시스템에 도움이 되었는가?"

그래서 보상은 예측 계층보다 액션 계층에 더 직접적으로 연결됩니다.

## 4. 가장 쉬운 이해 방식

파이프라인을 다음처럼 보면 이해가 쉽습니다.

```text
시스템 상태 -> CatBoost 위험도 예측 -> 에이전트 액션 -> 시스템 결과 -> 보상
```

예시:

1. 시스템이 메모리 압박이 커지는 노드를 관측합니다.
2. `CatBoost`가 향후 15분 내 failure 위험이 높다고 예측합니다.
3. 에이전트가 해당 노드에서 low-priority workload 하나를 다른 곳으로 옮기기로 결정합니다.
4. 노드는 안정 상태를 유지하고 중요한 SLA 태스크는 영향받지 않습니다.
5. 에이전트는 positive reward를 받습니다.

반대로 에이전트가 필요 이상으로 많은 workload를 이동시켜 오버헤드를 키웠다면, 낮은 reward 또는 penalty를 받아야 합니다.

## 5. 왜 예측만으로는 부족한가

예측은 "문제가 생길 수 있다"는 사실만 알려줍니다.

하지만 다음은 알려주지 못합니다.

- 어떤 액션을 취해야 하는지
- 언제 액션을 취해야 하는지
- 액션 비용이 감수할 만한 수준인지
- 반복 운영에서 어떤 액션이 더 나은지

시스템 프로젝트에서는 이 부분이 중요합니다. 좋은 클러스터 컨트롤러는 단순 예보가 아니라 trade-off를 수행해야 합니다.

## 6. 이 프로젝트에서 "지속적"이라는 말의 의미

교수님이 말한 "에이전트는 지속적이어야 한다"는 의미는, 컨트롤러가 일회성 분류기가 아니라 반복적으로 동작하는 루프여야 한다는 뜻에 가깝습니다.

1. 현재 workload와 node 상태를 관측한다
2. 학습된 모델로 risk를 계산한다
3. 액션을 선택한다
4. 다음 상태를 기다린다
5. 실제 결과를 보고 reward를 계산한다
6. 이 과정을 반복한다

이것이 제어 시스템 관점의 프로젝트 구조입니다.

중요한 점은 처음부터 완전한 production RL 스택을 구현해야 한다는 뜻이 아니라는 것입니다. 핵심은 프로젝트를 "시간이 흐르며 반복되는 의사결정 시스템"으로 정의하는 것입니다.

## 7. 교수님이 말한 "보상"의 실제 의미

보상은 액션의 품질을 숫자로 표현한 점수입니다.

Positive 사례:

- 노드 failure를 막음
- SLO 또는 SLA 위반을 방지함
- 리소스 낭비를 줄임
- 에너지 효율을 개선함
- latency를 안정적으로 유지함

Negative 사례:

- 불필요한 migration을 발생시킴
- scheduling delay를 증가시킴
- 중요한 태스크를 축출함
- throughput을 떨어뜨림
- 얻는 이득에 비해 control overhead가 너무 큼

보상은 예측 정확도보다 실제 프로젝트 목표를 반영해야 합니다.

## 8. 보상 함수 예시

failure 예방 시나리오라면, 간단한 보상 함수는 다음처럼 둘 수 있습니다.

```text
reward
= + 10 * failure_avoided
-   3 * migration_cost
-   4 * latency_penalty
-   8 * sla_violation
-   2 * unnecessary_action
```

이 식은 예시일 뿐이지만 구조는 분명합니다.

- 좋은 결과에는 보상
- 해로운 부작용에는 패널티
- 쓸데없는 개입에도 패널티

## 9. 멀티 에이전트는 어디에 들어가는가

교수님은 에이전트끼리 직접 대화하지 않아도 유기적으로 동작해야 한다고도 했습니다.

보통 이것은 다음을 의미합니다.

- 에이전트들이 같은 환경 상태를 공유한다
- 공통 신호를 보고 필요한 정보를 추론한다
- 명시적 메시지 전달보다 시스템 상태를 통해 간접적으로 협력한다

이 프로젝트에서는 예를 들어 다음처럼 잡을 수 있습니다.

- Predictor agent: failure risk score를 생성
- Placement agent: 현재 클러스터 상태와 risk score를 바탕으로 workload 배치 결정
- Recovery 또는 eviction agent: 압박이 임계치를 넘을 때만 개입

이 에이전트들은 채팅 채널이 없어도 됩니다. 대신 다음과 같은 공유 관측값으로 협력할 수 있습니다.

- machine utilization
- 예측된 failure risk
- 최근 placement 결정
- priority와 SLA 레벨

## 10. 캡스톤 설명 구조로 정리하면

비ML 청중에게 가장 깔끔한 설명은 다음 3단 구조입니다.

### Layer 1. Prediction

`CatBoost`가 시계열 및 리소스 feature를 기반으로 near-term failure 또는 overload risk를 예측합니다.

### Layer 2. Decision

위험도가 높을 때 어떤 액션을 취할지 에이전트가 결정합니다.

### Layer 3. Evaluation

그 액션이 시스템 전체에 도움이 되었는지 reward function으로 평가합니다.

이 구조는 "에이전트가 그냥 모델이다"라고 말하는 것보다 훨씬 방어가 쉽습니다.

## 11. 오브젝트는 어떻게 구체화할 것인가

미팅 메모에 따르면 object를 더 구체화해야 합니다.

예를 들어 다음과 같은 정의가 좋습니다.

- "공유 compute node에서 near-term workload failure를 intervention cost를 최소화하며 방지한다."

또는

- "클러스터에서 overload 관련 failure를 줄이되 latency와 scheduling efficiency를 유지한다."

이 정의는 단순히

- "failure를 예측한다"

라고 말하는 것보다 훨씬 낫습니다.

예측은 도구이고, 시스템의 목표는 운영 성능 개선입니다.

## 12. 이 저장소 기준의 구체적 예시

현재 저장소 맥락에 맞추면 다음처럼 설명할 수 있습니다.

- Observation:
  - 태스크별 CPU, 메모리 이력
  - node utilization
  - priority
  - scheduling class
- Predictor:
  - `CatBoost`가 `5m`, `15m`, `30m`, `45m`, `60m` 안의 failure 확률을 출력
- Agent action:
  - no-op
  - 과열된 node에 새로운 workload를 배치하지 않음
  - low-priority workload를 이동
  - node가 붕괴 직전이면 low-priority workload를 축출
- Reward:
  - 낮은 overhead로 failure를 막으면 positive
  - 액션 비용이 이득보다 크면 negative

## 13. 리뷰어에게 한 문장으로 설명하면

"왜 CatBoost를 학습한 뒤에도 에이전트가 필요한가?"라는 질문에는 다음처럼 답하면 됩니다.

`CatBoost`는 앞으로 무슨 일이 일어날지를 알려주고, 에이전트는 그에 대해 무엇을 할지 결정하며, 보상 시스템은 그 결정이 반복 운영에서 실제로 클러스터를 개선했는지 평가합니다.

## 14. 구현 범위는 어떻게 조절할 것인가

캡스톤에서는 다음 단계 정도로 구현 범위를 잡는 것이 현실적입니다.

1. predictor를 학습한다
2. 작은 action space를 정의한다
3. 측정 가능한 reward를 정의한다
4. 과거 데이터 위에서 시뮬레이션 또는 replay를 수행한다
5. prediction metric뿐 아니라 system-level outcome으로 액션을 비교한다

이 정도면 너무 큰 RL 구현으로 바로 가지 않으면서도 전체 아이디어를 충분히 보여줄 수 있습니다.
